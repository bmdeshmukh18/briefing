# Requirements Document

## Introduction

NSE Pulse is a personal, auto-updating NSE (National Stock Exchange of India) market briefing website. A structured daily briefing is generated by ChatGPT, emailed to a Gmail inbox, parsed by an n8n automation pipeline into strict JSON, and committed to a GitHub repository. A static website hosted on GitHub Pages (served via a Cloudflare subdomain) reads these JSON files to display each trading day's briefing across clean, responsive sections. Over time the system accumulates a historical data layer (Nifty trend, FII/DII flows, market breadth, sector rotation) and automatically tracks whether the previous session's next-session outlook was correct.

The product is private and personal today, but its architecture intentionally supports a future transition to a paid public product (authentication and paywall). This document defines the requirements for the MVP scope (briefing display, last 30 days of history, Nifty and FII/DII charts, and prediction accuracy tracking) while capturing the data contract and access-model requirements needed to keep later phases stable.

The single most important artifact is the per-day JSON data contract: every part of the website reads from it, and the automation pipeline writes to it. Field names in this contract must remain stable once live.

## Glossary

- **System**: The complete NSE Pulse product, comprising the Ingestion_Pipeline, the Prediction_Tracker, the Data_Store, and the NSE_Pulse_Site.
- **NSE_Pulse_Site**: The static, framework-free (HTML/CSS/JavaScript) website hosted on GitHub Pages that renders briefings, charts, and the accuracy tracker.
- **Data_Store**: The GitHub repository holding version-controlled JSON files: per-day briefing files at `/data/briefings/YYYY-MM-DD.json`, a master `/data/index.json`, and an aggregated `/data/history.json`.
- **Briefing_File**: A single per-day JSON file conforming to the Per-Day JSON Schema, located at `/data/briefings/YYYY-MM-DD.json`.
- **Index_File**: The master file `/data/index.json` listing every available trading-day date and metadata for date navigation.
- **History_File**: The aggregated file `/data/history.json` containing time-series data used to render historical charts.
- **Per-Day JSON Schema**: The fixed JSON structure (defined in Requirement 1) that every Briefing_File conforms to.
- **Ingestion_Pipeline**: The n8n workflow that watches Gmail for the briefing email, invokes the Briefing_Parser, and commits the resulting Briefing_File to the Data_Store.
- **Briefing_Parser**: The OpenAI-API-based parsing step within the Ingestion_Pipeline that converts the email body into JSON conforming to the Per-Day JSON Schema.
- **Briefing_Email**: The daily email generated by ChatGPT with subject line "NSE Market Briefing" containing the market briefing content.
- **Prediction_Tracker**: The n8n workflow that fetches the actual Nifty 50 close, compares it against the predicted scenario ranges from the prior session's outlook, and writes the verification result back to the Briefing_File.
- **Outlook**: The next-session forecast section of a Briefing_File, containing scenarios with probabilities and price ranges, support and resistance levels, and key watch items.
- **Prediction_Result**: The portion of a Briefing_File recording the verified outcome of that day's Outlook (actual close, matched scenario, and accuracy tag).
- **Accuracy_Tag**: A classification of a verified prediction with one of the values "Correct", "Partial", or "Wrong".
- **Accuracy_Tracker_View**: The NSE_Pulse_Site section that displays a calendar of verified predictions and a rolling accuracy percentage.
- **Date_Navigator**: The NSE_Pulse_Site control that lets a Viewer select which trading day's briefing to display.
- **Viewer**: A person accessing the NSE_Pulse_Site to read briefings.
- **Trading_Day**: A date on which the NSE conducts a regular trading session.
- **Market_Holiday**: A weekday on which the NSE does not conduct a trading session.
- **Nifty50**: The Nifty 50 index, identified at the data source by the symbol `^NSEI`.
- **FII**: Foreign Institutional Investors.
- **DII**: Domestic Institutional Investors.

## Requirements

### Requirement 1: Per-Day JSON Data Contract

**User Story:** As the site developer, I want every daily briefing stored in a single fixed JSON structure, so that the website, the parser, and the prediction tracker all read and write a stable, predictable contract.

#### Acceptance Criteria

1. THE Data_Store SHALL store each Trading_Day's briefing as one Briefing_File located at the path `/data/briefings/YYYY-MM-DD.json`, where `YYYY-MM-DD` is the ISO-8601 calendar date of the session.
2. THE Briefing_File SHALL contain the top-level objects `meta`, `summary`, `triggers`, `deep_dive`, `outlook`, and `prediction_result`.
3. THE `meta` object SHALL contain the field `date` whose value equals the ISO-8601 date in the Briefing_File's filename, the field `session_status` whose value is one of "Trading" or "Holiday", and the field `market_tone` whose value is either a non-empty string of at most 60 characters or `null`.
4. THE `summary` object SHALL contain the fields `nifty50` (with `close` and `change_pct`), `sensex` (with `close` and `change_pct`), `broader_market` (with `smallcap_pct` and `midcap_pct`), `breadth` (with `nifty500_advances` and `nifty500_declines`), `sectors` (with `leaders`, `laggards`, and `count_advanced`), `key_gainers`, `key_losers`, `institutional_flows` (with `fii_net_cr` and `dii_net_cr`), and `macro` (with `brent_crude` and `india_10y_yield`).
5. WHERE a numeric field within the `summary` or `macro` objects has a value present, THE Briefing_File SHALL represent that value as a JSON number.
6. THE `triggers` object SHALL contain the fields `domestic` and `global`, each holding a list of strings.
7. THE `deep_dive` object SHALL contain the field `full_text` whose value is either a string or `null`.
8. THE `outlook` object SHALL contain the fields `base_case`, `scenarios`, `support_levels`, `resistance_levels`, and `key_watch`, where `scenarios` is a list of 1 to 5 objects and each object contains `name` (a string of at most 60 characters), `probability` (a JSON number from 0 to 100), `range_low`, and `range_high`.
9. THE objects within `outlook.scenarios` SHALL each satisfy the invariant that `range_low` is less than or equal to `range_high`.
10. THE `prediction_result` object SHALL contain the fields `actual_close`, `actual_change_pct`, `matched_scenario`, `accuracy_tag`, and `verified_at`, where `accuracy_tag` is one of "Correct", "Partial", "Wrong", or `null`, `verified_at` is an ISO-8601 date-time value or `null`, and `actual_close`, `actual_change_pct`, and `matched_scenario` are each either their respective JSON value type or `null`.
11. WHERE a schema field has no available value, THE Briefing_File SHALL set that field to JSON `null` rather than omitting the field.
12. THE Per-Day JSON Schema field names defined in this requirement SHALL remain unchanged across all releases after the first production briefing is committed.

### Requirement 2: Index and History Data Files

**User Story:** As the site developer, I want a master index of dates and an aggregated history file, so that the website can offer date navigation and historical charts without reading every individual briefing file.

#### Acceptance Criteria

1. THE Index_File SHALL contain a list, ordered in ascending calendar-date order (earliest date first), of every date for which a Briefing_File exists in the Data_Store.
2. WHEN a Briefing_File whose date is not already present in the Index_File is committed to the Data_Store, THE Ingestion_Pipeline SHALL add that date to the Index_File while preserving the ascending calendar-date ordering.
3. THE History_File SHALL contain, per available Trading_Day and ordered in ascending calendar-date order (earliest date first), the Nifty50 close, the Nifty50 change percentage, the FII net flow in crore, the DII net flow in crore, and the breadth advance and decline counts.
4. WHEN a Briefing_File whose Trading_Day is not already present in the History_File is committed to the Data_Store, THE Ingestion_Pipeline SHALL append that day's time-series values to the History_File while preserving the ascending calendar-date ordering.
5. IF the Index_File or History_File does not exist when the Ingestion_Pipeline attempts to update it, THEN THE Ingestion_Pipeline SHALL create the file before adding the new entry.
6. IF a Briefing_File is committed for a date already present in the Index_File, THEN THE Ingestion_Pipeline SHALL leave the Index_File's existing date entries unchanged so that no duplicate date is added.
7. IF a Briefing_File is committed for a Trading_Day whose values are already present in the History_File, THEN THE Ingestion_Pipeline SHALL replace that day's existing time-series values rather than appending a duplicate entry.

### Requirement 3: Daily Briefing Ingestion via Email

**User Story:** As the site owner, I want the daily briefing email automatically parsed and stored, so that the website updates each day without manual effort.

#### Acceptance Criteria

1. WHEN an email with the subject line "NSE Market Briefing" arrives in the watched Gmail inbox, THE Ingestion_Pipeline SHALL invoke the Briefing_Parser on that email's body within 5 minutes of the email's arrival.
2. WHEN the Ingestion_Pipeline invokes the Briefing_Parser on a Briefing_Email, THE Briefing_Parser SHALL convert the email body into JSON conforming to the Per-Day JSON Schema within 120 seconds.
3. WHERE a field value is absent from the Briefing_Email content, THE Briefing_Parser SHALL set the corresponding schema field to `null`.
4. WHEN the Briefing_Parser produces JSON that conforms to the Per-Day JSON Schema and contains a non-null `meta.date` value, THE Ingestion_Pipeline SHALL commit that JSON to the Data_Store as a Briefing_File named for the ISO-8601 date taken from the `meta.date` field.
5. IF a Briefing_File already exists for the briefing date, THEN THE Ingestion_Pipeline SHALL overwrite the existing Briefing_File with the newly parsed content.
6. IF the Briefing_Parser fails to produce parseable JSON from the Briefing_Email, or does not complete within 120 seconds, THEN THE Ingestion_Pipeline SHALL send a parse-failure notification identifying the affected email's subject and arrival timestamp to the site owner and SHALL leave the Data_Store unchanged.
7. IF the Briefing_Parser produces parseable output that does not conform to the Per-Day JSON Schema, or whose `meta.date` value is `null` or absent, THEN THE Ingestion_Pipeline SHALL send a schema-validation-failure notification identifying the affected email's subject and arrival timestamp to the site owner and SHALL leave the Data_Store unchanged.
8. IF the commit of a Briefing_File to the Data_Store fails, THEN THE Ingestion_Pipeline SHALL retry the commit up to 3 additional times before treating the commit as failed.
9. IF all commit attempts for a Briefing_File fail, THEN THE Ingestion_Pipeline SHALL send a commit-failure notification identifying the affected briefing date to the site owner and SHALL leave the Data_Store unchanged.

### Requirement 4: Briefing Display Sections

**User Story:** As a Viewer, I want each day's briefing shown in clear, labeled sections, so that I can quickly read the full market picture.

#### Acceptance Criteria

1. WHEN a Viewer opens the NSE_Pulse_Site without specifying a date, THE NSE_Pulse_Site SHALL display the Briefing_File for the latest date listed in the Index_File.
2. THE NSE_Pulse_Site SHALL display a header containing the NSE Pulse logo and the Date_Navigator.
3. WHEN the NSE_Pulse_Site displays a Briefing_File, THE NSE_Pulse_Site SHALL display a hero strip containing exactly four cards showing, respectively, the Nifty50 close and change percentage, the Sensex close and change percentage, the FII net flow in crore, and the DII net flow in crore.
4. WHEN the NSE_Pulse_Site displays a Briefing_File, THE NSE_Pulse_Site SHALL display a Market Snapshot section showing the breadth advance and decline counts, the sector leaders and laggards, and the market tone.
5. WHEN the NSE_Pulse_Site displays a Briefing_File, THE NSE_Pulse_Site SHALL display a Movers section containing a key gainers table and a key losers table.
6. WHEN the NSE_Pulse_Site displays a Briefing_File, THE NSE_Pulse_Site SHALL display a Macro section showing the Brent crude price, the India 10-year yield, the domestic triggers, and the global triggers.
7. WHEN the NSE_Pulse_Site displays a Briefing_File, THE NSE_Pulse_Site SHALL display a Deep Dive section containing the `deep_dive.full_text` content in a collapsible container.
8. WHEN the NSE_Pulse_Site displays a Briefing_File, THE NSE_Pulse_Site SHALL display an Outlook section showing, for each scenario, the scenario name and its probability rendered as a bar whose filled length is proportional to that scenario's probability value, together with the support levels, the resistance levels, and the key watch items.
9. WHEN a Viewer activates a collapsible section control, THE NSE_Pulse_Site SHALL toggle that section's content between visible and hidden, switching it to the opposite of its current visibility state.

### Requirement 5: Date Navigation

**User Story:** As a Viewer, I want to browse briefings from previous trading days, so that I can review past market sessions.

#### Acceptance Criteria

1. WHEN the NSE_Pulse_Site loads, THE Date_Navigator SHALL present every date listed in the Index_File ordered from most recent to oldest, with the most recent date listed first.
2. WHEN a Viewer selects a date in the Date_Navigator, THE NSE_Pulse_Site SHALL load and display the Briefing_File for the selected date within 3 seconds of a successful retrieval from the Data_Store.
3. THE Date_Navigator SHALL restrict selectable dates to those present in the Index_File and SHALL NOT allow selection of any date absent from the Index_File.
4. IF a Viewer requests a date that is present in the Index_File but has no corresponding Briefing_File in the Data_Store, THEN THE NSE_Pulse_Site SHALL display a message stating that no briefing is available for that date and SHALL keep the Date_Navigator available for selecting another date.
5. IF a Viewer requests a date that is not present in the Index_File through a direct URL or bookmark, THEN THE NSE_Pulse_Site SHALL block access to that briefing and SHALL display a message stating that no briefing is available for that date.
6. IF a Viewer requests a date through a direct URL or bookmark that is not a valid ISO-8601 calendar date, THEN THE NSE_Pulse_Site SHALL block access to that briefing and SHALL display a message stating that no briefing is available for that date.

### Requirement 6: Historical Charts

**User Story:** As a Viewer, I want historical trend charts, so that I can see how the market has moved over recent sessions.

#### Acceptance Criteria

1. WHEN the NSE_Pulse_Site loads, THE NSE_Pulse_Site SHALL render the historical charts using data read from the History_File.
2. THE NSE_Pulse_Site SHALL provide tabbed chart views for Nifty Trend, FII/DII Flows, Market Breadth, and Sectors.
3. WHEN a Viewer selects a chart tab, THE NSE_Pulse_Site SHALL display the chart for the selected tab and hide the other chart tabs' content.
4. THE NSE_Pulse_Site SHALL render every historical chart covering the most recent 30 available Trading_Days.
5. WHERE fewer than 30 Trading_Days of data exist in the History_File, THE NSE_Pulse_Site SHALL render the charts using all available Trading_Days.
6. IF a data point in the History_File is `null`, THEN THE NSE_Pulse_Site SHALL omit that data point from the chart and SHALL render the remaining non-null data points without substituting a value for the omitted point.
7. WHEN the historical charts first render, THE NSE_Pulse_Site SHALL display the Nifty Trend chart and hide the other chart tabs' content.
8. IF the History_File cannot be retrieved or cannot be parsed as valid JSON, THEN THE NSE_Pulse_Site SHALL display a message stating that historical chart data is unavailable.
9. WHERE the History_File contains zero Trading_Days, THE NSE_Pulse_Site SHALL display a message stating that no historical data is available yet in place of the charts.

### Requirement 7: Automated Prediction Accuracy Verification

**User Story:** As the site owner, I want yesterday's outlook automatically checked against the actual close, so that the site can report how accurate the predictions are.

#### Acceptance Criteria

1. WHEN the Prediction_Tracker runs on a Trading_Day, THE Prediction_Tracker SHALL fetch the actual Nifty50 closing value from the configured primary market data source using a request timeout of 30 seconds.
2. WHEN the Prediction_Tracker obtains the actual Nifty50 close, THE Prediction_Tracker SHALL determine, for each scenario in the prior session's Outlook, whether the actual Nifty50 close is within that scenario's range, where within range means `range_low` is less than or equal to the actual Nifty50 close and the actual Nifty50 close is less than or equal to `range_high`.
3. IF two or more scenarios in the Outlook share the highest probability value, THEN THE Prediction_Tracker SHALL treat the first such scenario in the Outlook's scenario list order as the highest-probability scenario.
4. IF the actual Nifty50 close is within the range of the highest-probability scenario, THEN THE Prediction_Tracker SHALL set the `accuracy_tag` to "Correct" and set `matched_scenario` to that scenario's name.
5. IF the actual Nifty50 close is within the range of one or more scenarios but not within the range of the highest-probability scenario, THEN THE Prediction_Tracker SHALL set the `accuracy_tag` to "Partial" and set `matched_scenario` to the name of the matching scenario having the highest probability, resolving any tie by selecting the first such scenario in the Outlook's scenario list order.
6. IF the actual Nifty50 close falls outside the range of every scenario in the Outlook, THEN THE Prediction_Tracker SHALL set the `accuracy_tag` to "Wrong" and set `matched_scenario` to `null`.
7. WHEN the Prediction_Tracker determines an Accuracy_Tag, THE Prediction_Tracker SHALL write `actual_close`, `actual_change_pct`, `matched_scenario`, `accuracy_tag`, and the verification timestamp into the `prediction_result` of the corresponding Briefing_File.
8. IF the configured primary market data source returns an error or does not respond within the 30-second request timeout, THEN THE Prediction_Tracker SHALL fetch the actual Nifty50 close from the configured backup market data source using a request timeout of 30 seconds.
9. IF both the primary and backup market data sources return an error or fail to respond within their 30-second request timeouts, THEN THE Prediction_Tracker SHALL leave all `prediction_result` fields as `null` and SHALL send an error notification to the site owner indicating that the actual Nifty50 close could not be retrieved.

### Requirement 8: Prediction Accuracy Display

**User Story:** As a Viewer, I want to see the track record of predictions, so that I can judge how reliable the outlook has been.

#### Acceptance Criteria

1. THE Accuracy_Tracker_View SHALL display a calendar covering the most recent 30 Trading_Days, in which each Trading_Day with a non-null `accuracy_tag` is marked with a visually distinct indicator corresponding to its Accuracy_Tag value ("Correct", "Partial", or "Wrong").
2. THE Accuracy_Tracker_View SHALL display a rolling accuracy percentage calculated over the most recent 30 Trading_Days that have a non-null `accuracy_tag`, as the count of those Trading_Days tagged "Correct" divided by the count of those Trading_Days, multiplied by 100 and rounded to one decimal place.
3. WHERE at least one Trading_Day within the most recent 30 Trading_Days has a non-null `accuracy_tag` and none of those Trading_Days is tagged "Correct", THE Accuracy_Tracker_View SHALL display a message stating that no verified correct predictions are available yet in place of the rolling accuracy percentage.
4. WHEN a Viewer selects a marked day in the Accuracy_Tracker_View calendar, THE NSE_Pulse_Site SHALL display that day's Briefing_File.
5. WHERE no Trading_Day within the most recent 30 Trading_Days has a non-null `accuracy_tag`, THE Accuracy_Tracker_View SHALL display a message stating that no verified predictions are available yet in place of the rolling accuracy percentage.

### Requirement 9: Responsive Layout

**User Story:** As a Viewer, I want the site to work on both desktop and handheld devices, so that I can read briefings anywhere.

#### Acceptance Criteria

1. WHILE the viewport width is at least 768 pixels, THE NSE_Pulse_Site SHALL render the briefing sections in a two-column layout consisting of exactly two side-by-side columns.
2. WHILE the viewport width is from 320 pixels up to but not including 768 pixels, THE NSE_Pulse_Site SHALL render the briefing sections in a single-column layout.
3. WHILE the viewport width is from 320 pixels up to but not including 768 pixels, THE NSE_Pulse_Site SHALL render the Deep Dive, Macro, and Outlook sections as collapsible sections that are collapsed by default.
4. WHILE the viewport width is at least 768 pixels, THE NSE_Pulse_Site SHALL render the Deep Dive, Macro, and Outlook sections expanded by default.
5. WHILE the viewport width is from 320 pixels up to but not including 768 pixels, THE NSE_Pulse_Site SHALL render all briefing content within the viewport width without requiring horizontal scrolling.
6. WHILE a Viewer scrolls the briefing content, THE NSE_Pulse_Site SHALL keep the header fixed and fully visible at the top of the viewport without obscuring the briefing content that scrolls beneath it.

### Requirement 10: Hosting and Delivery

**User Story:** As the site owner, I want the site served as a free static deployment on a custom subdomain, so that there is zero hosting maintenance.

#### Acceptance Criteria

1. THE NSE_Pulse_Site SHALL be deployable as static HTML, CSS, and JavaScript files served by GitHub Pages, with no server-side code executed to generate responses.
2. WHEN a Viewer loads a page of the NSE_Pulse_Site or selects a date in the Date_Navigator, THE NSE_Pulse_Site SHALL retrieve the required briefing, index, and history data from the Data_Store through client-side requests issued from the Viewer's browser, without invoking a backend server.
3. WHEN a Viewer requests the configured Cloudflare subdomain, THE NSE_Pulse_Site SHALL return its pages and assets at that subdomain.
4. IF a request to the configured Cloudflare subdomain fails to resolve or is unreachable, THEN THE NSE_Pulse_Site SHALL remain accessible at the underlying GitHub Pages default domain.
5. THE NSE_Pulse_Site SHALL be built using only HTML, CSS, and JavaScript, without a front-end framework.

### Requirement 11: Access Model (Private Now, Paid Later)

**User Story:** As the site owner, I want the site private now but architected for a future paywall, so that I can later sell access without re-architecting.

#### Acceptance Criteria

1. THE NSE_Pulse_Site SHALL read all data from the Data_Store through a single configurable data-access layer.
2. THE NSE_Pulse_Site SHALL classify as gated content any Briefing_File other than the most recent Trading_Day's Briefing_File, all History_File-based charts, and the `deep_dive` section content.
3. WHERE an access-control configuration flag is disabled, THE NSE_Pulse_Site SHALL grant access to all briefing content without requiring authentication.
4. WHERE an access-control configuration flag is enabled, WHILE a Viewer is authenticated, THE NSE_Pulse_Site SHALL grant that Viewer access to the gated content.
5. IF an access-control configuration flag is enabled and an unauthenticated Viewer requests gated content, THEN THE NSE_Pulse_Site SHALL withhold the gated content and SHALL display an authentication prompt.
6. IF a Viewer's authentication attempt is unsuccessful, THEN THE NSE_Pulse_Site SHALL keep the gated content withheld and SHALL display a message indicating that authentication failed.
7. THE Per-Day JSON Schema field names SHALL remain unchanged regardless of the access-control configuration flag's state.

### Requirement 12: Market Holiday and Non-Trading-Day Handling

**User Story:** As a Viewer, I want the site to behave correctly on days with no trading session, so that the absence of a briefing is clear and the accuracy stats stay correct.

#### Acceptance Criteria

1. IF a requested date is a Market_Holiday or a non-trading weekend day, THEN THE NSE_Pulse_Site SHALL display a message stating that no trading session occurred on that date and SHALL NOT display any briefing content sections for that date.
2. THE Date_Navigator SHALL exclude any date that has no Briefing_File in the Data_Store from the selectable dates.
3. WHEN the Prediction_Tracker is scheduled to run on a Market_Holiday or a non-trading weekend day, THE Prediction_Tracker SHALL skip verification for that date and SHALL leave that date's prediction_result fields unchanged.
4. IF the Prediction_Tracker skips verification because the date is a Market_Holiday or a non-trading weekend day, THEN THE Prediction_Tracker SHALL NOT send an error notification to the site owner.
5. THE Accuracy_Tracker_View SHALL exclude Market_Holiday dates and non-trading weekend days from both the calendar markings and the rolling accuracy percentage calculation.

### Requirement 13: Missing and Malformed Data Handling

**User Story:** As a Viewer, I want the site to remain usable when some data is missing or malformed, so that one bad field does not break the whole page.

#### Acceptance Criteria

1. IF a displayed schema field is `null`, THEN THE NSE_Pulse_Site SHALL render a visible placeholder indicator (for example, a dash) that is distinct from an empty value in place of that field's value.
2. IF a Briefing_File cannot be retrieved from the Data_Store or cannot be parsed as valid JSON, THEN THE NSE_Pulse_Site SHALL display a message stating that the briefing for that date could not be loaded while keeping the header and Date_Navigator interactive.
3. IF the Index_File cannot be retrieved from the Data_Store or cannot be parsed as valid JSON, THEN THE NSE_Pulse_Site SHALL display a message stating that the briefing list is unavailable.
4. IF a list field such as `key_gainers`, `key_losers`, `support_levels`, or `key_watch` is empty, THEN THE NSE_Pulse_Site SHALL display, within that field's section, a message stating that no entries are available for that section.
5. IF an individual field or section value is `null`, missing, or malformed, THEN THE NSE_Pulse_Site SHALL render all remaining non-null, well-formed fields and sections of the page.

### Requirement 14: SEO Metadata and Admin Annotations

**User Story:** As the site owner, I want SEO metadata and a private per-day annotation area, so that the site is discoverable later and I can keep private notes per day.

#### Acceptance Criteria

1. WHEN the NSE_Pulse_Site renders a briefing, THE NSE_Pulse_Site SHALL set the page title to include the briefing date and the market tone.
2. WHEN the NSE_Pulse_Site renders a briefing, THE NSE_Pulse_Site SHALL set the page meta description to a string of at most 160 characters that includes the briefing date and the market tone.
3. IF the briefing date or the market tone needed for the page title or the page meta description is `null` or unavailable, THEN THE NSE_Pulse_Site SHALL set the page title and the page meta description using only the available values rather than blocking the page.
4. THE Briefing_File SHALL support an optional `admin_notes` field whose value, when present, is a string of at most 5,000 characters.
5. WHERE the admin-view configuration flag is disabled, THE NSE_Pulse_Site SHALL exclude the `admin_notes` content from both the rendered page and the page source delivered to the Viewer.
6. WHERE the admin-view configuration flag is enabled and the `admin_notes` field has a non-empty value, THE NSE_Pulse_Site SHALL display the `admin_notes` content for the displayed Trading_Day.
7. WHERE the admin-view configuration flag is enabled and the `admin_notes` field is `null` or empty, THE NSE_Pulse_Site SHALL render the briefing without an admin notes value and without blocking the page.
